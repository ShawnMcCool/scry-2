Phase 1

Setup — clearing state & suspending workers

All five steps are sequential DB calls or synchronous GenServer calls. Total time: milliseconds.

Step 1 — DB delete
Repo.delete_all(IngestionState.Snapshot) — wipes the crash-recovery singleton row so IngestRawEvents starts fresh on restart.
Step 2 — DB delete
Repo.delete_all(Events.EventRecord) — deletes all domain events. Projections are not deleted here — each projector's rebuild! owns that in Phase 3.
Step 3 — DB bulk update
Repo.update_all(..., set: [processed: false]) on all mtga_logs_events rows. One SQL statement, fast regardless of row count.
single UPDATE ... SET
Step 4 — GenServer.call
suspend_ingest_raw_events_checkpointing() — tells IngestRawEvents to stop writing IngestionState to the DB after each event. Without this, reingest would do ~8 000 singleton upserts with zero crash-recovery value.
optimization: skips ~8k DB writes
Step 5 — 5 × GenServer.call
suspend_all_projectors() — suspends all 5 projectors (Matches, Drafts, Mulligans, Ranks, Economy). While suspended, projectors consume {:domain_event, ...} PubSub messages as fast no-ops instead of projecting. Without this, every domain event produced during retranslation would be live-projected — then immediately thrown away by rebuild! in Phase 3.
optimization: eliminates 5× redundant projection

Phase 2 · Detail

Per-event cost breakdown

Every one of ~8 000 raw events runs this work. Costs multiply linearly.

DB operations per event

N × INSERT — domain_events
One row per domain event produced. Most raw events produce 0–1 domain events. GreToClientEvent (mulligan) can produce several.
1 × UPDATE — mtga_logs_events
mark_processed!(id) — sets processed = true and processed_at on the raw event row.
0 × UPSERT — ingestion_state (suppressed)
Without checkpointing suspension: IngestionState.persist! would write a singleton row after every event. Suspended → eliminated.

In-process work per event

JSON decode (conditional)
Only GreToClientEvent and RankGetCombinedRankInfo events require JSON decode before translation. All others skip this.
Translation (always)
IdentifyDomainEvents.translate/3 — pure function, CPU-only. For unrecognized event types this is a near-instant lookup miss. For handled types it parses the JSON payload.
5 × PubSub no-op (suppressed)
Without projector suspension: each Events.append! broadcast would wake all 5 projector GenServers to do a full DB fetch + project + watermark write. Suspended → message consumed, no DB work.

Sync point sync_ingest_raw_events()

Calls :sys.get_state(IngestRawEvents) — a synchronous GenServer call. The call cannot be handled until all prior handle_info messages in the mailbox are processed. This is how the retranslation loop becomes blocking: we wait here until every raw event has been translated and persisted.

Each projector runs rebuild!() in its own Task.Supervisor task — not through its GenServer mailbox. They read non-overlapping slices of domain_events (different event_type filters) and write to entirely separate tables. No coordination needed; no conflicts possible. The 5 tasks execute in parallel, bounded only by SQLite's write concurrency (WAL mode).

Loop continues until no more rows returned. Final watermark set to max event ID in log.
Why cursor-based instead of loading all events?
Bounded memory — never loads more than 500 rehydrated structs at once. Crash-safe — the watermark after each batch means a partial rebuild can resume via catch_up!() without re-projecting already-processed events.
Phase 4

Resume all projectors

— resume_all_projectors() — 5 × GenServer.call

Sets paused: false on each projector GenServer. From this point, {:domain_event, id, type} broadcasts from live MTGA ingestion are processed normally. Projectors are now at the correct watermark (set by rebuild!), so any events that arrived during the reingest are handled in order.

What was optimized — before vs after

Before

🐢 Sequential projector rebuild
Enum.each(projectors, & &1.rebuild!()) — 5 projectors rebuilt one after another. Total time = sum of all 5.
🐢 IngestionState persisted per event
IngestionState.persist!/1 called after every raw event — ~8 000 singleton DB upserts during retranslation with zero crash-recovery value.
🐢 5× redundant live projection during retranslation
Every domain event produced during retranslation was immediately live-projected by all 5 projectors — then completely discarded by rebuild!. ~2 800 domain events × 5 projectors = ~14 000 wasted DB operations.
💥 KeyError on hot code reload
Adding paused key to projector state while GenServers were live caused state.paused to raise KeyError on old in-flight state. Projectors crashed and the pipeline stalled.

After

⚡ Parallel projector rebuild
Task.Supervisor.async_stream(..., timeout: :infinity) — 5 projectors rebuild concurrently. Total time ≈ slowest projector (Matches).
⚡ Checkpointing suspended during reingest
suspend_ingest_raw_events_checkpointing() before broadcast, resume_ingest_raw_events_checkpointing() after drain. Zero DB writes for IngestionState during retranslation.
⚡ Projectors suspended during retranslation
suspend_all_projectors() / resume_all_projectors(). Projectors consume PubSub messages as fast no-ops. Zero wasted projection work — rebuild! does it cleanly once, from scratch.
✅ Defensive state access
Map.get(state, :paused, false) and Map.put(state, :paused, ...) instead of struct-update syntax. Works with both old state %{} (hot reload) and new state %{paused: false}.

Remaining bottleneck

Retranslation (Phase 2) is still fundamentally sequential — one GenServer, one event at a time. This cannot be parallelized without breaking event ordering. The only path to faster retranslation is reducing per-event cost: batching DB writes (e.g. multi-row INSERTs), avoiding JSON re-decode (the raw JSON is re-decoded by the translator on every event), or reducing the number of raw events that produce domain events (filtering truly irrelevant event types at ingest time).
