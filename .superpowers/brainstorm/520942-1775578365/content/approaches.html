Event Viewer — Component Architecture

How should the reusable pieces be structured?

A

LiveComponent Kit

Reusable LiveComponents — each filter, the event list, and the detail view are separate Phoenix.LiveComponent modules. The /events page composes them. Other pages embed whichever components they need.

Pros

  • Each component owns its own state and events
  • Embedding = <.live_component module={EventList} .../>
  • Components can communicate via send/send_update

Cons

  • LiveComponent state management adds complexity
  • Cross-component coordination (filters → list) requires message passing
  • LiveComponents are heavier than function components
B

Context Query API + Function Components

Smart context, thin views. The Events context gets a rich query API (filter by type, time, correlation, text search). Function components render results. Each LiveView composes queries + components as needed. Filters live in URL params.

Pros

  • Follows existing project pattern (Matches, Drafts, Cards)
  • URL-driven state — bookmarkable, shareable, survives reload
  • Function components are simple, testable, no hidden state
  • Any LiveView can call Events.list_events(filters) and render results

Cons

  • Filter UI code may be duplicated across pages (mitigated by shared function components)
  • Each host LiveView must wire up filter → query → assign
C

Hybrid — Query API + One LiveComponent Shell

Best of both. Rich Events query API (like B), but wrap the full "event explorer" (filters + list + pagination) in a single EventExplorer LiveComponent. Embedding pages drop in the component with preset filters. Detail view is a separate page.

Pros

  • One-line embedding: <.live_component module={EventExplorer} filters={%{...}} />
  • Query API still available for custom compositions
  • Filter + list coordination stays encapsulated

Cons

  • LiveComponent adds weight when only the query API is needed
  • Preset filters vs. user filters need clear ownership rules
  • More abstraction layers than the project currently uses
